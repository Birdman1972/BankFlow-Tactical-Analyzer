# 意見回饋 API 整合方案評估與決策

## 目標
建立可擴充、可控且符合隱私要求的回饋收集機制，支援離線暫存與重試。

## 方案比較

### 1) GitHub Issues API
**優點**
- 省後端維護成本
- 可直接進入開發流程（Issue 追蹤）

**缺點**
- 需使用者具備 GitHub 帳號與登入流程
- 不適合一般使用者回饋
- 可能暴露專案內部資訊

### 2) Google Forms
**優點**
- 建置快速、免後端
- 視覺化表單管理方便

**缺點**
- 客製化與整合性弱
- 隱私控管不易（儲存位置與存取權限）
- 不易加入版本/平台等結構化資料

### 3) Vercel Serverless API（自建）
**優點**
- 完全可控（資料格式、驗證、節流、存取權）
- 易於與版本檢查、離線佇列整合
- 可逐步擴充（後台、統計、匯出）

**缺點**
- 需要維運與部署
- 需設計儲存策略與保留政策

## 決策
選擇 **Vercel Serverless API** 作為回饋收集方案。

## 理由
- 符合隱私與資料安全需求，可避免收集敏感資訊。
- 便於整合版本資訊與平台資訊。
- 支援離線暫存與重試機制，提升可靠性。
- 可在不影響 UI 的情況下持續擴充後端能力。

## 隱私與安全策略
- 只收集必要欄位（類型/標題/內容/版本/平台/時間）。
- 客戶端與伺服端皆進行輸入驗證與長度限制。
- 伺服端加上節流與大小限制，避免濫用。

## 離線策略
- Client 端以 IndexedDB 為主要佇列。
- 若 IndexedDB 不可用，降級為 localStorage。
- 提供重試與退避策略，避免短時間重送。
